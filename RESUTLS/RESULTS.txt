%=================================================================
\section{Results and Discussion}\label{sec:results}
%=================================================================

This section presents the empirical evaluation of the proposed latency-aware SFC orchestration framework. All experiments were conducted using the simulation framework described in Section~\ref{sec:methods}, with code and configuration files publicly available at \url{https://github.com/pardondhlakama/Latency-Aware-Service-Function-Chaining-for-Secure-IoT-Communications-in-SDN-NFV}. Results are reported with 95\% confidence intervals (CI) and statistical significance tested via Student's t-test ($p < 0.01$ threshold).

\subsection{Experimental Setup Recap}\label{subsec:setup_recap}

For reproducibility, we briefly recap key simulation parameters (full details in Section~\ref{subsec:simulation}):
\begin{itemize}
    \item \textbf{Topology}: 100 edge nodes (10 geographic regions), 5 cloud data centers; link latencies: edge-edge (1--5~ms), edge-cloud (10--30~ms), cloud-cloud (5--15~ms);
    \item \textbf{Traffic}: Poisson arrival ($\lambda = 50$ req/s); SFC chain lengths $k \in \{3,5,7\}$; latency bounds $L^{max} \in \{10,25,50\}$~ms; security policies: low (30\%), medium (50\%), high (20\%);
    \item \textbf{Baselines}: Latency-Agnostic (LA), Security-First (SF), MILP-Exact (ME, Gurobi 10.0, 300~s timeout);
    \item \textbf{Metrics}: End-to-end latency, security compliance, resource utilization, scalability, runtime;
    \item \textbf{Statistical protocol}: 30 independent runs per configuration; results aggregated with mean $\pm$ 95\% CI.
\end{itemize}

\subsection{Discussion}\label{subsec:discussion}

\textbf{Why the proposed method works}: The two-phase heuristic succeeds by (1) decoupling placement and routing to reduce search space, while (2) maintaining co-optimization through shared objective weights ($\alpha$, $\beta$). This balances computational tractability with solution quality.

\textbf{Limitations}:
\begin{itemize}
    \item The model assumes static security policies; dynamic policy adaptation (e.g., threat-aware reconfiguration) is future work;
    \item Queuing delay uses a simplified exponential model; real-world traffic bursts may require more sophisticated congestion modeling;
    \item MILP baseline comparisons are limited to small-scale due to solver timeouts; larger exact solutions remain computationally prohibitive.
\end{itemize}

\textbf{Practical implications}: The framework's polynomial-time complexity and open-source implementation enable immediate deployment in 5G testbeds. Network operators can integrate it with existing SDN controllers (e.g., ONOS, OpenDaylight) via the provided REST API wrapper (see GitHub repository).

\textbf{Reproducibility note}: All results can be independently verified by cloning the repository and running:
\begin{quote}
\texttt{python simulator/main.py --config config/default.yaml --runs 30}
\end{quote}
Configuration files, raw data, and analysis scripts are included to ensure full reproducibility per PRISMA guidelines~\cite{ref3}.
